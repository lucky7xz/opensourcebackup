OpensourceBackup Control Plane — B1–B7

⚠️ Aktuelle Version (B1–B7): Keine Authentifizierung. Nicht für Produktionseinsatz verwenden.

Was ist OpensourceBackup?

OpensourceBackup ist eine Backup Control Plane — eine zentrale Plattform zur Verwaltung, Überwachung und Koordination von Backup-Jobs auf vielen Systemen gleichzeitig.

Deine Systeme → Backup-Agent → Control Plane → Storage
                                     ↑
                               Du steuerst hier

Was du damit tust: Systeme registrieren, Repositories definieren, Policies anlegen, Jobs überwachen, Snapshots verwalten.

Starten

git clone https://github.com/cerberus8484/opensourcebackup.git
cd opensourcebackup

make dev-up        # PostgreSQL + Redis (Docker)
make migrate-up    # Datenbanktabellen anlegen
make run           # → http://localhost:8080

curl http://localhost:8080/health
# → {"status":"ok"}

Systeme /v1/systems

Ein System ist ein zu sicherndes Gerät: Server, VM, Datenbank oder Endgerät.

System anlegen

curl -X POST http://localhost:8080/v1/systems \
  -H "Content-Type: application/json" \
  -d '{
    "Hostname": "web-server-01.example.com",
    "OS": "Ubuntu 22.04",
    "RiskClass": "critical",
    "Tags": {"env": "prod", "team": "backend"}
  }'
FeldPflichtBeschreibung
Hostname✅Eindeutiger Hostname
OS—Betriebssystem
RiskClass—standard (Default) oder critical
Tags—Freie Key-Value-Metadaten (JSON)

Systeme abrufen & verwalten

GET    /v1/systems          # alle Systeme
GET    /v1/systems/{id}     # ein System
PUT    /v1/systems/{id}     # aktualisieren
DELETE /v1/systems/{id}     # löschen → HTTP 204

Repositories /v1/repositories

Ein Repository ist ein Backup-Speicherziel (S3-Bucket, MinIO, ZFS-Dataset).

curl -X POST http://localhost:8080/v1/repositories \
  -H "Content-Type: application/json" \
  -d '{
    "Type": "restic",
    "Location": "s3:mein-bucket/backups/web-01",
    "EncryptionMode": "aes256",
    "ObjectLockEnabled": true
  }'
FeldPflichtBeschreibung
Type✅restic, borg, pgbackrest, velero
Location✅Pfad oder URL zum Storage
EncryptionMode—Verschlüsselungsmodus
ObjectLockEnabled—WORM-Schutz gegen Ransomware

Policies /v1/policies

Eine Policy definiert: was wird gesichert, wann, mit welcher Engine, wie lange aufbewahrt.

curl -X POST http://localhost:8080/v1/policies \
  -H "Content-Type: application/json" \
  -d '{
    "Name": "nightly-full-backup",
    "Engine": "restic",
    "Includes": ["/home", "/etc", "/var/www"],
    "Excludes": ["/home/*/.cache"],
    "Schedule": "0 2 * * *",
    "Retention": {"daily": 7, "weekly": 4, "monthly": 12}
  }'
FeldPflichtBeschreibung
Name✅Eindeutiger Name
Engine✅restic, borg, pgbackrest, velero
Schedule—Cron-Ausdruck: 0 2 * * * = täglich 02:00
Retention—Aufbewahrungsregeln: {"daily":7,"weekly":4}

Cron-Beispiele

ScheduleBedeutung
0 2 * * *Täglich 02:00 Uhr
0 2 * * 0Wöchentlich, Sonntags 02:00
0 */6 * * *Alle 6 Stunden
@dailyEinmal täglich

Jobs /v1/jobs

Ein Job ist eine ausgeführte oder geplante Backup-Instanz.

curl -X POST http://localhost:8080/v1/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "SystemID": "uuid-des-systems",
    "PolicyID": "uuid-der-policy",
    "Status": "pending"
  }'
StatusBedeutung
pendingWarte auf Agent
runningAgent führt Backup aus
successBackup erfolgreich
failedBackup fehlgeschlagen
warningBackup mit Warnungen

Snapshots /v1/snapshots

Ein Snapshot ist das Ergebnis eines erfolgreichen Backup-Jobs.

curl -X POST http://localhost:8080/v1/snapshots \
  -H "Content-Type: application/json" \
  -d '{
    "JobID": "uuid-des-jobs",
    "RepositoryID": "uuid-des-repositories",
    "EngineSnapshotID": "abc123def456",
    "Hostname": "web-server-01",
    "Paths": ["/home", "/etc"],
    "ChecksumStatus": "verified"
  }'

Scheduler & Dead-Man's-Switch

Der Scheduler lädt beim Start alle Policies mit Cron-Schedule und plant automatisch Backup-Jobs. Der Dead-Man's-Switch prüft alle 5 Minuten ob alle Policies fristgerecht ausgeführt wurden:

{"level":"WARN","msg":"dead-man: overdue job detected",
 "policy_name":"nightly-full-backup",
 "last_job_at":"2024-01-01T02:00:00Z"}

Häufige Fragen

Warum keine Anmeldung?

Auth kommt in B9. Aktuelle Version ist nur für Entwicklung geeignet — nicht für Produktion.

DB-Verbindung prüfen

curl http://localhost:8080/health
# {"status":"ok"}   → DB erreichbar
# HTTP 503          → DB nicht erreichbar

Alles zurücksetzen

make migrate-down  # Tabellen löschen
make migrate-up    # Neu anlegen

Sicherheitshinweise

  • Keine Produktionsdaten in der aktuellen Version speichern (keine Auth)
  • DATABASE_URL enthält Credentials — niemals in Logs ausgeben
  • .env.local niemals committen — ist in .gitignore
  • Jeder API-Endpunkt gibt Security Headers zurück (CSP, X-Frame-Options, HSTS, …)
